# 统一点位管理系统设计方案（最新版）

## 目标与原则

1. 以 `SPoint` 为核心，统一管理协议点位、反射遥测点位、配置点位。
2. 所有协议点位一律通过 `SProtocolPoint` 作为基础，再派生协议具体点位（如 `SModbusPoint`、`SCanbusPoint`、`SDidioPoint`）。
3. 点位定义采用 A+D 混合模式：构造函数 + 选项模式（Functional Options）。
4. 优先复用 `c_default` 预定义点位与数据访问配置，减少重复与误配。
5. 创建后数据不可变；系统启动时进行完整性验证（必填字段必须设置）。
6. 保持业务层取值方式与接口不变；仅优化定义方式与结构分层。

## 分层结构

```text
SPoint（基础点位信息）
└── SProtocolPoint（协议点位基础，统一持有 DataAccess）
    ├── SModbusPoint（Modbus 协议点位）
    ├── SCanbusPoint（CAN 总线协议点位）
    └── SDidioPoint（DIDO 协议点位）

SReflectPoint（反射遥测点位，持有 *SPoint + 方法名）
SConfigPoint（配置点位，持有 *SPoint + 配置约束字段）
```

## 核心结构

```go
// SPoint 点位元数据
type SPoint struct {
    Key          string
    Name         string
    ValueType    c_enum.EValueType
    Group        *SPointGroup
    Unit         string
    Desc         string
    Sort         int
    Min          int64
    Max          int64
    Precise      uint8
    Hidden       bool
    ValueExplain []*SFieldExplain

    // 警戒逻辑（可选）
    Trigger func(value interface{}) (bool, c_enum.EAlarmLevel, error)
}

// SProtocolPoint 协议点位基础
type SProtocolPoint struct {
    *SPoint
    DataAccess *SDataAccess // 必填：数据访问配置
}

// SModbusPoint Modbus 协议点位
type SModbusPoint struct {
    *SProtocolPoint
    Addr uint16

    // 覆盖字段（如需与基础行为不同）
    ValueExplain []*c_base.SFieldExplain

    // 告警触发（可选）
    Trigger func(value interface{}) (bool, c_enum.EAlarmLevel, error)
}

// SReflectPoint 反射遥测点位
type SReflectPoint struct {
    *SPoint
    MethodName string
}

// SConfigPoint 配置点位（仅配置维度使用）
type SConfigPoint struct {
    *SPoint
    Required           bool
    Default            *string
    Regex              *string
    RegexFailedMessage *string
    Step               *float32
}
```

说明：
- 子结构体不重复实现接口方法，通过嵌套复用 `SPoint`/`SProtocolPoint` 行为。
- 协议点位一律通过 `SProtocolPoint` 管理 `DataAccess`，并在启动时校验不能为空。

## 点位定义规范（Modbus 为例）

### 字段顺序要求

1. `Addr`（Modbus 地址，放第一位）
2. `SProtocolPoint`（内含 `SPoint` 与 `DataAccess`）
3. `ValueExplain`（可选）
4. `Trigger`（可选）

### 构造函数与选项模式

- 基础构造（80% 场景）：
  - `NewModbusPoint`
  - `NewModbusPointWithUnit`
  - `NewModbusPointWithDesc`（推荐）
  - `NewModbusPointFromPreset`（推荐：复用 `c_default` 预设点位）

- 复杂场景（20%）：
  - `NewModbusPointExt(addr, With...options...)`
  - 选项函数：`WithKey/WithName/WithValueType/WithUnit/WithDesc/WithDataAccess/WithValueExplain/WithTrigger/WithGroup/WithMinMax/WithPrecise/WithHidden/WithPresetPoint`

示例：

```go
// 简单点位
var Year = c_proto.NewModbusPointWithDesc(30297, "Year", "年", c_enum.EUint16, "", "年", c_default.VDataAccessUInt16)

// 使用预设点位
var DCVoltage = c_proto.NewModbusPointFromPreset(0x1103, c_default.VPointDcVoltage, c_default.VDataAccessUInt16Scale01)

// 复杂点位（值解释/自定义访问/告警触发）
var OnOffCommand = c_proto.NewModbusPointExt(30314,
    c_proto.WithKey("OnOffCommand"),
    c_proto.WithName("开关机指令"),
    c_proto.WithValueType(c_enum.EUint16),
    c_proto.WithDesc("On/off command: 0- Shutdown, 1- Startup, 2- Standby"),
    c_proto.WithDataAccess(c_default.VDataAccessUInt16),
    c_proto.WithValueExplain([]*c_base.SFieldExplain{
        {Key: "0", Value: "关机",   Color: "#d9d9d9"},
        {Key: "1", Value: "已启动", Color: "#52c41a"},
        {Key: "2", Value: "待机",   Color: "#faad14"},
    }),
)
```

### 预定义配置优先

- 点位信息：优先使用 `c_default.VPoint*`（如 `VPointUa`、`VPointSOC`）。
- 数据访问：优先使用 `c_default.VDataAccess*`（如 `VDataAccessFloat32`、`VDataAccessUInt16ToBool`）。
- 避免冗余默认值：`Factor:1/Offset:0/ByteEndian:Big/WordOrder:HighLow` 均无需显式设置。
- 位域访问统一：`BitIndex/BitLength` 放前，`DataFormat: BitRange` 与 `ValueType` 匹配。

## 启动时验证

1. `SModbusPoint.SProtocolPoint` 必须非空。
2. `SProtocolPoint.SPoint` 必须非空。
3. `SProtocolPoint.DataAccess` 必须非空。
4. 校验失败阻止启动并输出可定位的错误信息（设备、点位 Key、原因）。

## 遥测获取统一逻辑

```go
// 统一遥测入口（示意）
func GetTelemetryValue(point c_base.IPoint, dev c_base.IDevice) (any, error) {
    switch p := point.(type) {
    case *c_base.SReflectPoint:
        return getTelemetryByReflect(p.MethodName, dev)
    case *c_base.SProtocolPoint:
        return getTelemetryFromCache(p, dev)
    default:
        return nil, errors.New("unsupported point type")
    }
}
```

说明：
- 反射点位通过方法名获取；协议点位从缓存读取。
- 驱动可实现 `GetTelemetryPoints()` 暴露关键遥测点位集合。

## 规则与最佳实践

- 点位 Key 在同一驱动文件中必须唯一。
- Modbus 任务按功能与地址连续性分组；设置合理 `CycleMill` 与 `Lifetime`。
- 使用 `ValueExplain` 统一值解释；布尔型 Key 使用 `"false"/"true"`。
- 告警触发统一使用 `Trigger` 函数或 `c_default` 预置触发器（Warn/Alert/Error）。
- 包级别定义点位，便于复用与启动期验证。

## 影响与兼容性

- 业务取值方式、接口定义保持不变；迁移仅影响点位“定义方式”。
- 通过构造函数与选项模式显著降低定义复杂度，提升一致性与可维护性。

## 验证清单

- 所有点位均采用构造函数/选项模式定义。
- 预定义 `c_default` 配置优先使用，无冗余默认值。
- 启动时验证通过，无空指针与缺失配置。
- 驱动关键遥测点位可通过统一入口读取。

# 统一点位管理系统设计方案

## 项目概述

本方案旨在为EMS项目创建一个统一的点位管理系统，解决当前系统中协议点位、遥测点位、配置点位管理分散、类型不统一的问题。通过结构体嵌套和类型判断实现统一管理，同时保持各协议类型的独立性和扩展性。

## 需求分析

### 1. 核心需求

1. **统一点位接口**：需要有一个可以定义modbus、canbus、dido之类的点位接口，用来定义协议中的查询方法
2. **缓存查询优化**：缓存可以用key来查询，但缓存返回的数据需要有该点位的结构化信息
3. **驱动遥测定义**：driver可以定义固定的telemetry，用来反射执行driver的get方法，获取数据
4. **遥测结构化信息**：telemetry需要有结构化信息，用来展示数据，还有创建自动化任务时使用telemetry进行对比
5. **配置信息结构体**：config配置信息结构体，一般是一些保存在字符串中的动态配置，需要创建一个config
6. **双重数据获取**：在driver中，希望telemetry既可以是反射执行方法获取的数据，也可以是直接从缓存中读取的
7. **协议点位定义**：协议点位一般都是直接定义的，比如@ammeter_demo_point_v.go
8. **代码化遥测**：telemetry现在是写在yaml文件中@build.yaml，现在准备使用代码来添加telemetry
9. **统一遥测获取**：通过telemetry获取数据时，分别通过反射或者缓存方式来获取值
10. **动态配置支持**：动态的配置类似：@pylon_tech_us108_config_s.go，会在BuildDescriptionFromYaml方法中添加到SDriverInfo
11. **点位缓存机制**：协议在通过点位查询到数据后，会缓存点位和value到SPointValue中
12. **系统架构参考**：整个系统还有@config-struct-tags.mdc、@c_base_device_i.go、@c_base_driver_i.go、@c_base_protocol_i_s.go需要参考
13. **统一结构管理**：希望所有协议、遥测、配置，都拥有统一的结构进行管理，但会有自己的扩展
14. **单一获取方式**：遥测只能通过一种方式来获取，要么是反射，要么是缓存，通过判断该点位是使用什么类型定义的
15. **不兼容YAML**：不需要兼容现在的yaml配置
16. **保留原有结构**：原有的SModbusPoint等结构，尽量保留
17. **多场景应用**：需要用到点位的地方有：协议、telemetry、配置结构体描述、对外的输出服务

### 2. 技术决策

1. **获取方式判断**：通过判断类型来实现，比如A是S的子结构体，判断是否是S类型，是不是A就能知道需要用什么方式来获取数据了
2. **结构体设计**：不希望统一的结构体，希望区分结构体，但结构体有嵌套的关系
3. **不向前兼容**：完全移除v1兼容性，直接使用新的统一结构体
4. **配置系统**：使用新的统一的结构体
5. **对外输出**：不用定义新的，比如A是反射的结构体，B是缓存的结构体（也就是协议的结构体），C是配置的结构体，S是公共的，那么对外输出时，只要读取S列表，就可以知道应该怎么输出了
6. **不可变设计**：所有结构体数据在创建后不允许修改，确保数据一致性
7. **启动时验证**：在系统启动时验证所有*SPoint是否设置，确保数据完整性

## 当前系统分析

### 1. 现有架构

#### 点位系统

- **基础接口**：`IPoint` - 定义了所有点位的通用接口，提供统一访问方式
- **基础实现**：`SPoint` - 点位元数据，包含基本信息
- **协议扩展**：`SModbusPoint`、`SCanbusPoint` - 包含 `*SPoint`，添加协议特定信息
- **配置点位**：`SConfigPoint` - 包含 `*SPoint`，用于配置结构体描述
- **反射点位**：`SReflectPoint` - 包含 `*SPoint`，用于反射遥测点位

#### 遥测系统

- 当前通过YAML文件定义遥测信息
- 通过反射调用 `Get{TelemetryKey}` 方法获取数据
- 遥测信息存储在 `SDriverInfo.Telemetry` 中

#### 配置系统

- 使用 `SFieldDefinition` 描述配置字段
- 通过 `BuildConfigStructFields` 方法解析配置结构体
- 支持动态标签解析（SHORT、JSON、YAML）

#### 缓存系统

- 协议读取数据后缓存到 `SPointValue` 中
- 通过 `IProtocolCacheValue` 接口提供缓存访问

### 2. 现有问题

1. **重复定义**：遥测信息在YAML和代码中重复定义
2. **类型不统一**：协议点位、遥测、配置使用不同的结构体
3. **获取方式单一**：遥测只能通过反射获取，不能从缓存获取
4. **扩展性不足**：新增协议类型需要创建新的点位结构体
5. **接口复杂性**：IPoint接口方法较多，但实际使用中可能不需要所有方法

## 设计方案

### 1. 基础点位结构体层次

```text
SPoint (基础点位信息)
├── SReflectPoint (反射遥测点位)
├── SProtocolPoint (协议点位基础)
│   ├── SModbusPoint (Modbus协议点位)
│   ├── SCanbusPoint (CAN总线协议点位)
│   └── SDidioPoint (DIDO协议点位)
└── SConfigPoint (配置点位)
```

### 3. IPoint接口设计说明

#### 3.1 保留IPoint接口的原因

1. **多态性支持**：不同协议的点位有不同的扩展字段，接口提供统一访问方式
2. **类型安全**：通过接口约束，确保所有点位都实现必要的方法
3. **扩展性**：未来新增点位类型时，只需实现IPoint接口
4. **接口隔离**：只暴露必要的点位信息，隐藏实现细节

#### 3.2 简化IPoint接口

```go
// IPoint 简化的点位接口，只包含核心方法
type IPoint interface {
    GetKey() string                            // 点位Key
    GetName() string                           // 名称
    GetValueType() c_enum.EValueType           // 值类型
    GetValueExplain(value any) (string, error) // 获取Value解释
    IsAlarmPoint() bool                        // 是否是告警点位
    TriggerAlarm(value any) (trigger bool, level c_enum.EAlarmLevel, err error) // 告警触发
}

// 注意：子结构体不需要重复实现IPoint接口方法
// 通过结构体嵌套，自动继承SPoint的方法实现
// 只有在需要覆盖方法时才重新实现
```

### 5. 协议点位结构调整

#### StatusExplain 字段移除说明

原有的 `StatusExplain func(value any) (text string, err error)` 字段已被移除，统一使用 `ValueExplain []*SFieldExplain` 字段来实现值解释功能。

**迁移规则**：
- 布尔值状态解释：`StatusExplainBool(value, "trueText", "falseText")` → `ValueExplain: []*SFieldExplain{{Key: "false", Value: "falseText"}, {Key: "true", Value: "trueText"}}`
- 数值状态解释：switch-case 逻辑 → 转换为对应的 ValueExplain 数组，Key 使用字符串格式的数值
- 颜色设置：根据状态类型设置合适的颜色（正常=绿色，保护=黄色，错误=红色等）

**Key 设置规范**：
- **布尔值类型** (`c_enum.EBool`)：Key 使用 `"false"` 和 `"true"`
- **数值类型** (`c_enum.EUint16`, `c_enum.EInt16` 等)：Key 使用字符串格式的数值，如 `"0"`, `"1"`, `"2"`
- **浮点类型** (`c_enum.EFloat32`, `c_enum.EFloat64`)：Key 使用字符串格式的数值，如 `"0.0"`, `"1.5"`

**层层查找逻辑**：
1. 优先使用协议点位自己的 ValueExplain
2. 然后使用 SProtocolPoint 的 ValueExplain  
3. 最后使用 SPoint 的 ValueExplain
4. 如果都没有，返回空切片

### 6. 启动时验证机制

```go
// ValidateAllPoints 验证所有点位的SPoint字段是否设置
func ValidateAllPoints() error {
    // 验证所有驱动中的点位定义
    deviceManager := GetDeviceManager()
    if deviceManager == nil {
        return errors.New("device manager not initialized")
    }
  
    var validationErrors []string
  
    // 遍历所有设备
    deviceManager.IteratorAllDevices(func(config *SDeviceConfig, device IDevice) bool {
        // 验证所有点位（遥测点位+协议点位）
        allPoints := device.GetPoints()
        for i, point := range allPoints {
            if point == nil {
                validationErrors = append(validationErrors, 
                    fmt.Sprintf("设备 %s 的点位[%d]为空", config.Id, i))
            }
        }
      
        // 验证配置点位
        configParam := device.GetConfigParam()
        if configParam != nil {
            configPoints, err := BuildConfigPoints(configParam)
            if err != nil {
                validationErrors = append(validationErrors, 
                    fmt.Sprintf("设备 %s 的配置点位解析失败: %v", config.Id, err))
            } else {
                for i, point := range configPoints {
                    if point.SPoint == nil {
                        validationErrors = append(validationErrors, 
                            fmt.Sprintf("设备 %s 的配置点位[%d]的SPoint字段未设置", config.Id, i))
                    }
                }
            }
        }
      
        return true // 继续遍历
    })
  
    if len(validationErrors) > 0 {
        return errors.New("点位验证失败: " + strings.Join(validationErrors, "; "))
    }
  
    return nil
}
```

// GetTelemetryValue 统一遥测数据获取
func GetTelemetryValue(point IPoint, instance IDevice) (any, error) {
    switch p := point.(type) {
    case *SReflectPoint:
        // 通过反射获取
        return getTelemetryByReflect(p.MethodName, instance)
    case *SProtocolPoint:
        // 从缓存获取
        return getTelemetryFromCache(p, instance)
    default:
        return nil, errors.New("unsupported point type")
    }
}

    // 使用现有的反射逻辑
    return getTelemetry(methodName, instance)
}

// getTelemetryFromCache 从缓存获取遥测数据
func getTelemetryFromCache(point *SProtocolPoint, instance IDevice) (any, error) {
    // 从协议缓存中获取数据
    if cache, ok := instance.(IProtocolCacheValue); ok {
        return cache.GetValue(point)
    }

## 总结

本方案通过创建分层的点位管理系统，实现了协议点位、遥测点位、配置点位的统一管理。通过结构体嵌套和类型判断，既保持了各协议类型的独立性，又实现了统一的数据获取和管理。方案具有良好的扩展性，可以轻松支持新的协议类型。

### 主要优势

1. **类型安全**：通过类型判断确定获取方式，避免配置错误
2. **统一管理**：所有点位通过统一接口管理，便于扩展
3. **灵活扩展**：新增协议类型只需继承SProtocolPoint

5. **不可变设计**：结构体数据不可变，确保数据一致性

### 关键改进
