# 统一点位管理系统设计方案

## 项目概述

本方案旨在为EMS项目创建一个统一的点位管理系统，解决当前系统中协议点位、遥测点位、配置点位管理分散、类型不统一的问题。通过结构体嵌套和类型判断实现统一管理，同时保持各协议类型的独立性和扩展性。

## 需求分析

### 1. 核心需求

1. **统一点位接口**：需要有一个可以定义modbus、canbus、dido之类的点位接口，用来定义协议中的查询方法
2. **缓存查询优化**：缓存可以用key来查询，但缓存返回的数据需要有该点位的结构化信息
3. **驱动遥测定义**：driver可以定义固定的telemetry，用来反射执行driver的get方法，获取数据
4. **遥测结构化信息**：telemetry需要有结构化信息，用来展示数据，还有创建自动化任务时使用telemetry进行对比
5. **配置信息结构体**：config配置信息结构体，一般是一些保存在字符串中的动态配置，需要创建一个config
6. **双重数据获取**：在driver中，希望telemetry既可以是反射执行方法获取的数据，也可以是直接从缓存中读取的
7. **协议点位定义**：协议点位一般都是直接定义的，比如@ammeter_demo_point_v.go
8. **代码化遥测**：telemetry现在是写在yaml文件中@build.yaml，现在准备使用代码来添加telemetry
9. **统一遥测获取**：通过telemetry获取数据时，分别通过反射或者缓存方式来获取值
10. **动态配置支持**：动态的配置类似：@pylon_tech_us108_config_s.go，会在BuildDescriptionFromYaml方法中添加到SDriverInfo
11. **点位缓存机制**：协议在通过点位查询到数据后，会缓存点位和value到SPointValue中
12. **系统架构参考**：整个系统还有@config-struct-tags.mdc、@c_base_device_i.go、@c_base_driver_i.go、@c_base_protocol_i_s.go需要参考
13. **统一结构管理**：希望所有协议、遥测、配置，都拥有统一的结构进行管理，但会有自己的扩展
14. **单一获取方式**：遥测只能通过一种方式来获取，要么是反射，要么是缓存，通过判断该点位是使用什么类型定义的
15. **不兼容YAML**：不需要兼容现在的yaml配置
16. **保留原有结构**：原有的SModbusPoint等结构，尽量保留
17. **多场景应用**：需要用到点位的地方有：协议、telemetry、配置结构体描述、对外的输出服务

### 2. 技术决策

1. **获取方式判断**：通过判断类型来实现，比如A是S的子结构体，判断是否是S类型，是不是A就能知道需要用什么方式来获取数据了
2. **结构体设计**：不希望统一的结构体，希望区分结构体，但结构体有嵌套的关系
3. **向后兼容**：保持原有的SModbusPoint等结构不变，但需要调整它们与新的统一系统的关系
4. **配置系统**：使用新的统一的结构体
5. **对外输出**：不用定义新的，比如A是反射的结构体，B是缓存的结构体（也就是协议的结构体），C是配置的结构体，S是公共的，那么对外输出时，只要读取S列表，就可以知道应该怎么输出了

## 当前系统分析

### 1. 现有架构

#### 点位系统

- **基础接口**：`IPoint` - 定义了所有点位的通用接口，提供统一访问方式
- **基础实现**：`SPoint` - 点位元数据，包含基本信息
- **协议扩展**：`SModbusPoint`、`SCanbusPoint` - 包含 `*SPoint`，添加协议特定信息
- **配置点位**：`SConfigPoint` - 包含 `*SPoint`，用于配置结构体描述
- **反射点位**：`SReflectPoint` - 包含 `*SPoint`，用于反射遥测点位

#### 遥测系统

- 当前通过YAML文件定义遥测信息
- 通过反射调用 `Get{TelemetryKey}` 方法获取数据
- 遥测信息存储在 `SDriverInfo.Telemetry` 中

#### 配置系统

- 使用 `SFieldDefinition` 描述配置字段
- 通过 `BuildConfigStructFields` 方法解析配置结构体
- 支持动态标签解析（SHORT、JSON、YAML）

#### 缓存系统

- 协议读取数据后缓存到 `SPointValue` 中
- 通过 `IProtocolCacheValue` 接口提供缓存访问

### 2. 现有问题

1. **重复定义**：遥测信息在YAML和代码中重复定义
2. **类型不统一**：协议点位、遥测、配置使用不同的结构体
3. **获取方式单一**：遥测只能通过反射获取，不能从缓存获取
4. **扩展性不足**：新增协议类型需要创建新的点位结构体
5. **接口复杂性**：IPoint接口方法较多，但实际使用中可能不需要所有方法

## 设计方案

### 1. 基础点位结构体层次

```text
SPoint (基础点位信息)
├── SReflectPoint (反射遥测点位)
├── SProtocolPoint (协议点位基础)
│   ├── SModbusPoint (Modbus协议点位)
│   ├── SCanbusPoint (CAN总线协议点位)
│   └── SDidioPoint (DIDO协议点位)
└── SConfigPoint (配置点位)
```

### 3. IPoint接口设计说明

#### 3.1 保留IPoint接口的原因

1. **多态性支持**：不同协议的点位有不同的扩展字段，接口提供统一访问方式
2. **类型安全**：通过接口约束，确保所有点位都实现必要的方法
3. **扩展性**：未来新增点位类型时，只需实现IPoint接口
4. **接口隔离**：只暴露必要的点位信息，隐藏实现细节

#### 3.2 简化IPoint接口

```go
// IPoint 简化的点位接口，只包含核心方法
type IPoint interface {
    GetKey() string                            // 点位Key
    GetName() string                           // 名称
    GetValueType() c_enum.EValueType           // 值类型
    GetValueExplain(value any) (string, error) // 获取Value解释
    IsAlarmPoint() bool                        // 是否是告警点位
    TriggerAlarm(value any) (trigger bool, level c_enum.EAlarmLevel, err error) // 告警触发
}

// 可选：如果需要更多信息，可以通过类型断言访问SPoint
func GetPointInfo(point IPoint) *SPoint {
    switch p := point.(type) {
    case *SReflectPoint:
        return p.SPoint
    case *SProtocolPoint:
        return p.SPoint
    case *SConfigPoint:
        return p.SPoint
    default:
        return nil
    }
}
```

### 6. 遥测数据获取逻辑

```go
// GetTelemetryValue 统一遥测数据获取
func GetTelemetryValue(point IPoint, instance IDevice) (any, error) {
    switch p := point.(type) {
    case *SReflectPoint:
        // 通过反射获取
        return getTelemetryByReflect(p.MethodName, instance)
    case *SProtocolPoint:
        // 从缓存获取
        return getTelemetryFromCache(p, instance)
    default:
        return nil, errors.New("unsupported point type")
    }
}

// getTelemetryByReflect 通过反射获取遥测数据
func getTelemetryByReflect(methodName string, instance IDevice) (any, error) {
    // 使用现有的反射逻辑
    return getTelemetry(methodName, instance)
}

// getTelemetryFromCache 从缓存获取遥测数据
func getTelemetryFromCache(point *SProtocolPoint, instance IDevice) (any, error) {
    // 从协议缓存中获取数据
    if cache, ok := instance.(IProtocolCacheValue); ok {
        return cache.GetValue(point)
    }
    return nil, errors.New("instance does not support cache access")
}
```

### 7. 驱动实现示例

#### 7.1 预定义配置使用规范

**优先使用预定义配置**：在创建点位时，应优先使用 `c_default` 包中的预定义配置，避免手动创建配置。

**预定义配置优势**：

1. **标准化**：使用经过验证的标准配置
2. **一致性**：确保相同类型点位使用相同配置
3. **维护性**：集中管理，便于更新和维护
4. **告警功能**：预定义点位已包含告警触发逻辑

**预定义配置类型**：

- **点位配置**：`VPointIa`、`VPointUa`、`VPointSoc` 等电力系统常用点位
- **数据访问配置**：`VDataAccessFloat32`、`VDataAccessUInt16` 等Modbus常用配置
- **点位分组**：`VPointGroupSystemBasic` 等预定义分组
- **告警触发**：预定义点位已包含告警触发函数

#### 7.2 使用预定义配置
