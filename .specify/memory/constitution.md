# EMS Plan 项目宪法

## 核心原则

### I. 插件化架构原则

所有核心功能必须通过插件系统实现。驱动（Drivers）、协议（Protocols）、存储（Storages）、推送（Push）、策略（Policy）均为可插拔模块，便于扩展与替换。每个插件都是独立的Go模块，通过接口通信，支持热插拔和动态配置。

**插件类型**：

- **设备驱动插件** (plug_drivers/): PCS、BMS、电表、消防控制、GPIO等
- **通信协议插件** (plug_protocols/): Modbus、CANBus、GPIO Sysfs
- **数据存储插件** (plug_storages/): InfluxDB、PebbleDB、TSDB、SQLite
- **策略插件** (plug_policy/): 储能策略、微电网策略

### II. 统一点位管理原则

以SPoint为核心，统一管理协议点位、反射遥测点位、配置点位。所有协议点位通过SProtocolPoint作为基础，派生具体协议点位。点位定义采用构造函数+选项模式，优先复用c_default预定义配置，创建后数据不可变，启动时进行完整性验证。

**核心结构层次**：

```text
SPoint（基础点位信息）
└── SProtocolPoint（协议点位基础，持有 DataAccess）
    ├── SModbusPoint（Modbus 协议点位）
    ├── SCanbusPoint（CAN 总线协议点位）
    └── SDidioPoint（DIDO 协议点位）

SReflectPoint（反射遥测点位，持有 *SPoint + MethodName）
SConfigPoint（配置点位，持有 *SPoint + 配置约束字段）
```

### III. 命名规范原则

严格遵循Golang类型缩写规范：接口I前缀、结构体S前缀、枚举E前缀、常量C前缀。文件命名使用类型后缀：*_i.go接口、*_s.go结构体、*_e.go枚举、*_f.go函数、*_c.go常量。所有公共方法、字段、接口和结构体都必须有清晰的中文注解。

**文件命名规范**：

- `*_i.go`: 接口定义 (Interface)
- `*_s.go`: 结构体定义 (Struct)
- `*_e.go`: 枚举定义 (Enum)
- `*_f.go`: 函数定义 (Function)
- `*_c.go`: 常量定义 (Constant)
- `*_test.go`: 测试文件

### IV. 代码质量原则

统一使用github.com/pkg/errors进行错误处理，使用c_log进行日志记录。所有公共方法、字段、接口和结构体都必须有简洁明了的中文注解，说明功能和用途。使用context.Context进行上下文传递，单例模式用于服务管理，接口分离原则。

**日志系统规范**：

- **系统日志**: `c_log.级别名(ctx, 参数...)`
- **业务日志**: `c_log.Biz级别名(ctx, 参数...)`
- **日志级别**: Debug, Info, Notice, Warning, Error, Critical, Panic, Fatal

### V. 测试驱动原则

系统启动时必须验证所有必填字段，确保数据完整性。提供测试模式功能，程序启动10秒后自动关闭，用于快速验证启动状态。所有服务都应该有对应的测试文件，使用接口进行依赖注入，便于单元测试。

**测试模式特性**：

- 自动关闭：程序启动10秒后自动关闭
- 完整启动：执行完整的系统初始化流程
- 日志记录：记录完整的启动和关闭日志
- 安全退出：使用标准的关闭信号处理机制

### VI. API设计原则

一个接口一个文件，文件命名格式为[module]_[action].go。RESTful API设计，操作类接口使用空响应结构体，查询类接口返回具体数据。非必填参数使用指针类型，统一使用GoFrame验证标签，使用s_db_model中的字段常量进行数据库操作。

**响应结构体设计**：

- **操作类接口**：使用空响应结构体，成功时返回空结构体，失败时通过err返回
- **查询类接口**：返回具体的数据结构，时间字段使用*time.Time类型

## 技术架构约束

### 项目结构要求

```
ems-plan/
├── common/                    # 共享库 - 所有模块间共享的接口和类型
│   ├── c_base/               # 基础接口和类型定义
│   ├── c_device/             # 设备相关实现
│   ├── c_proto/              # 协议相关
│   ├── c_enum/               # 枚举定义和常量
│   └── c_default/            # 预定义配置
├── services/                 # 服务层 - 核心业务服务
│   ├── s_db/                 # 数据库服务
│   ├── s_driver/             # 驱动服务  
│   └── s_storage/            # 存储服务
├── plugins/                  # 插件系统
│   ├── plug_drivers/         # 设备驱动插件
│   ├── plug_protocols/       # 协议插件
│   ├── plug_storages/        # 存储插件
│   └── plug_policy/          # 策略插件
├── application/              # 主应用程序
└── @cpp/                     # C++ 扩展库
```

### 技术栈要求

- **Go 1.24+**：使用最新稳定版Go语言
- **GoFrame v2**：统一的Web框架和ORM
- **SQLite**：轻量级数据库存储
- **Context**：上下文传递和生命周期管理
- **CGO**：C++扩展支持

### 依赖管理要求

- **github.com/pkg/errors**：错误处理
- **github.com/shockerli/cvt**：类型转换
- **gopkg.in/yaml.v3**：YAML配置解析

### 构建环境要求

- **开发环境构建**：必须使用-tags=dev参数
- **生产环境构建**：不要使用dev标签
- **强制启动**：所有启动命令默认包含--force=true参数
- **测试模式**：支持--test=true参数，10秒后自动关闭

### 数据存储要求

- **配置数据库**：使用SQLite存储设备配置、协议配置、系统设置
- **时序数据库**：支持Prometheus TSDB存储设备遥测数据
- **字段常量使用**：必须使用s_db_model中定义的字段常量，避免硬编码字符串

## 开发工作流

### 代码审查要求

1. **命名一致性**：遵循项目的命名规范
2. **类型安全**：确保泛型使用正确
3. **错误处理**：统一使用github.com/pkg/errors
4. **接口实现**：确保所有接口都有对应实现
5. **日志使用**：正确使用c_log进行日志记录
6. **注解规范**：确保所有公共方法、字段、接口和结构体都有适当的注解

### 服务开发流程

1. **服务层开发**：在services/目录下创建独立模块
2. **插件开发**：在plugins/目录下实现插件功能
3. **应用层开发**：在application/目录下实现Web接口
4. **数据库服务**：遵循s_db开发规范，使用字段常量进行数据库操作

### 开发检查清单

#### 数据库实体创建

- [ ] 在 `s_db_init_f.go` 中添加 SQL 表创建语句
- [ ] 在 `s_db_model/` 中创建模型文件
- [ ] 在 `s_db_basic_interface.go` 中定义服务接口
- [ ] 在 `internal/impl/` 中实现服务
- [ ] 在 `s_db_export_f.go` 中添加导出函数
- [ ] 字段常量规范：通用字段在 `s_db_model_const_c.go` 中定义，表特有字段在各自模型文件中定义
- [ ] 确保所有方法都有适当的日志记录
- [ ] 测试 CRUD 操作的正确性

#### 点位定义

- [ ] 所有点位使用构造函数/选项模式定义
- [ ] 优先使用 `c_default` 预定义配置
- [ ] 无冗余默认值（Factor:1, Offset:0 等）
- [ ] 点位 Key 在驱动中唯一
- [ ] 启动时验证通过，无空指针
- [ ] 业务取值方法继续正常工作

### 质量门禁

- 所有代码必须通过编译检查
- 启动时验证必须通过，无空指针与缺失配置
- 业务取值方法必须继续正常工作
- 所有点位使用构造函数/选项模式定义
- 优先使用c_default预定义配置

### 注意事项

#### 代码质量要求

- 所有接口都应该有对应的实现
- 使用泛型时确保类型安全
- 错误处理要统一使用 `github.com/pkg/errors`
- 单例模式要确保线程安全
- 配置结构体要支持 YAML 序列化
- 所有公共方法、字段、接口和结构体都必须有清晰的中文注解

#### 命名规范要求

- 严格遵循前缀规则：接口 `I`、结构体 `S`、枚举 `E`、常量 `C`
- 文件名一致性：必须使用正确的类型后缀
- 避免命名冲突：确保类型名在包内唯一
- 语义化命名：所有命名都应具有明确的语义
- 向后兼容：修改现有类型名时要考虑向后兼容性

#### 构建标签要求

- 开发环境构建：必须使用 `-tags=dev` 参数
- 生产环境构建：不要使用 `dev` 标签
- 构建脚本：确保构建脚本正确设置标签参数
- 配置文件：检查 `config.yaml` 中的 `extra: "-tags=dev"` 设置

## 治理规则

### 宪法权威性

本宪法是项目的最高指导原则，所有开发活动都必须遵循宪法规定。任何违反宪法的代码或设计都必须立即修正。

### 修改程序

宪法修改需要：

1. 详细说明修改原因和影响范围
2. 更新相关模板文件和规则文件
3. 确保所有依赖文件保持一致性
4. 记录修改历史和版本变更

### 合规检查

所有PR和代码审查必须验证是否符合宪法要求。使用.cursor/rules/目录下的规则文件进行运行时开发指导。复杂性必须被充分论证，任何偏离标准实践的设计都需要明确说明理由。
